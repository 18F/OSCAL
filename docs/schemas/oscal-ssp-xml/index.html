Schema Reference - XML System Security Plan Model

Schema download oscal_ssp_schema.xsd

Schema version 1.0.0-milestone2

Short name oscal-ssp

XML namespace http://csrc.nist.gov/ns/oscal/1.0

The OSCAL Control SSP format can be used to describe the information typically specified in a system security plan, such as those defined in NIST SP 800-18.

The root of the OSCAL System Security Plan (SSP) format is system-security-plan.

A single line of an address.

This element appears inside: address.

This element appears inside: person and org.

  • city element [0 or 1], with text contents

  • state element [0 or 1], with text contents

    State, province or analogous geographical region for mailing address

  • postal-code element [0 or 1], with text contents

    Postal or ZIP code for mailing address

  • country element [0 or 1], with text contents

    Country for mailing address

Annotation: annotation element

A name/value pair with optional explanitory remarks.

This element appears inside: party, person, org, role, responsible-party, system-characteristics, system-information, leveraged-authorization, authorization-boundary, network-architecture, data-flow, system-implementation, user, component, service, interconnection, inventory-item, implemented-component, implemented-requirement, responsible-role, and by-component.

The annotation element has the following contents:

  • name attribute (required), type NCName
  • id attribute (optional), type string
  • ns attribute (optional), type string
  • valuevalue attribute (optional), type string

    Indicates the value of the characteristic.

  • remarks element [0 or 1], as prose content: paragraphs, lists etc.

    Additional commentary on the parent item.

Remarks

Characteristics permit the deployment and management of arbitrary controlled values for any purpose useful to an application or implementation scenario.

The value of a characteristic is a simple scalar value or list of values.

The lexical composition of characteristics may be constrained by declarations including matching to regular expressions or by declaring a known datatype.

Because characteristics are often used as selectors or identifiers for OSCAL operations, their values can be expected frequently to be normalized (e.g., with respect to whitespace) in use; however, this is application defined.

Availability Impact Level: availability-impact element

The expected level of impact resulting from the disruption of access to or use of information or the information system.

This element appears inside: information-type.

The availability-impact element has the following contents (in order):

  • prop elements [0 to ∞], with text contents

    A value with a name, attributed to the containing control, part, or group.

  • base element [1], type string

    The prescribed base (Confidentiality, Integrity, or Availability) security impact level.

    The value must be one of the following:

    • fips-199-low: A 'low' sensitivity level as defined in FIPS-199.
    • fips-199-moderate: A 'moderate' sensitivity level as defined in FIPS-199.
    • fips-199-high: A 'high' sensitivity level as defined in FIPS-199.
  • selected element [0 or 1], type string

    The selected (Confidentiality, Integrity, or Availability) security impact level.

    The value must be one of the following:

    • fips-199-low: A 'low' sensitivity level as defined in FIPS-199.
    • fips-199-moderate: A 'moderate' sensitivity level as defined in FIPS-199.
    • fips-199-high: A 'high' sensitivity level as defined in FIPS-199.
  • adjustment-justification element [0 or 1], as prose content: paragraphs, lists etc.

    If the selected security level is different from the base security level, this contains the justificaiton for the change.

Back matter: back-matter element

A collection of citations and resource references.

This element appears inside: system-security-plan.

The back-matter element has the following contents (in order):

  • citation elements [0 to ∞], with element contents

    A citation to resources, either external or internal (by means of internal cross-reference).

  • resource elements [0 to ∞], with element contents

    A resource associated with the present document.

Remarks

Provides a collection of identified citation and resource objects that can be referenced by a link with a rel value of "reference" and an href value that is a fragment "#" followed by a reference to a citation identifier. A citation can reference a resource by specifying a local target value using the same fragment-identifier approach.

  • The following is a contrived example to show the use of link, citation, and resource.

    <oscal>
            
<link rel="reference" href="#citation1">My citation</link>
            ...snip...
            
<back-matter>
               
<citation id="citation1">
                  
<target>#resource1</target>
               </citation>
               
<resource id="resource1">
                  
<rlink href="https://example.org/some-resource"></rlink>
               </resource>
            </back-matter>
         </oscal>

Citation: citation element

A citation to resources, either external or internal (by means of internal cross-reference).

This element appears inside: back-matter.

The citation element has the following contents (in order):

  • id attribute (required), type string
  • target elements [0 to ∞], with text contents

    An address for retrieval of a citation

  • title element [0 or 1], with mixed text and (inline prose) element contents

    A title for display and navigation

  • desc element [0 or 1], with text contents

    A short textual description

  • doc-id elements [0 to ∞], with text contents

    A document identifier qualified by an identifier type.

  • prop elements [0 to ∞], with text contents

    A value with a name, attributed to the containing control, part, or group.

  • Any element (in a foreign namespace)
Remarks

The structure of a citation is not intended to be a full biblographic representation of a cited resource. Instead, it is expected that matching to a source of bibliograpical information will be handled by use of a doc-id. Use of a canonical identifier such as a DOI is recommended for any citation.

Within this element, a generic title can cite a title. If citation information takes the form of a bibliographic entry, then desc can be used to provide it.

Component: component element

A defined component that can be part of an implemented system.

This element appears inside: system-implementation.

The component element has the following contents (in order):

  • idid attribute (required), type NCName

    A unique identifier for a component.

  • namename attribute (required), type string

    The component's human-readable name.

  • component-type attribute (required), type string

    The value may be locally defined, or one of the following:

    • software: Any software, operating system, or firmware
    • hardware
    • service: A service that may provide APIs.
    • policy
    • process
    • proceedure
    • plan
    • guidance: Any guideline or recommendation
    • standard
    • validation
  • description element [1], as prose content: paragraphs, lists etc.

    A description of the component, including information about its function.

  • prop elements [0 to ∞], with text contents

    A value with a name, attributed to the containing control, part, or group.

    Some specific property names can be assigned to the component at large, or can be defined when a component is associated with an inventory-item within the implemented-component construct.

  • annotation elements [0 to ∞], with element contents

    A name/value pair with optional explanitory remarks.

    Some specific annotation names can be assigned to the component at large, or can be defined when a component is associated with an inventory-item within the implemented-component construct.

  • link elements [0 to ∞], with mixed text and (inline prose) element contents

    A reference to a local or remote resource

  • status element [1], with element contents

    Describes the operational status of the system.

  • responsible-role elements [0 to ∞], with element contents

    Defines a role that has responsibility for the component.

    A role defined in this way can be associated with a party or parties here, or where the component is referenced (e.g., inventory-item, implemented-requirement/by-component).

    The roles "maintainer" or "provider" can be used to define the supplier. In such cases party-id(s) can be used to associate the providing organization. For products, services, and APIs, this is the vendor and/or retailer information. For policies, processes, plans, standards, and guidance, this is the authoring organization.

  • remarks element [0 or 1], as prose content: paragraphs, lists etc.

    Additional commentary on the parent item.

Remarks

Components may be products, services, application programming interface (APIs), policies, processes, plans, guidance, standards, or other tangible items that enable security and/or privacy.

The type indicates which of these component types is represented.
